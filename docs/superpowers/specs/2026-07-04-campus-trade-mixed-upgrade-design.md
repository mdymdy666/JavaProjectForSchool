# 校园二手交易平台混合实训升级版设计

## 1. 背景与目标

当前项目已经具备 Vue3 前端、Spring Boot 后端、MySQL 建表脚本和可运行的演示接口，但核心业务仍以 `/api/demo/**` 内存数据和单页前端为主，距离课程要求中的“功能完善、界面友好、安全可靠、支持 MySQL 与 Redis”的校园 C2C 二手交易系统还有明显差距。

本设计采用“混合实训升级版”：用户、商品、订单、消息、后台管理等核心模块按真实系统方式设计并逐步落到 MySQL；Redis 用于验证码、缓存、浏览量、热度、限流和 token 辅助状态；推荐、敏感词、举报等扩展能力先做轻量可演示版本。目标是在答辩周期内做出能跑通、能讲清、代码结构不粗糙的完整实训项目。

## 2. 总体原则

- 前端使用 Vue3 + Vite + TypeScript，采用路由、API 层、状态管理和组件化结构，不再把主要业务堆在 `App.vue`。
- 后端使用 Spring Boot + Spring Security + JWT + MyBatis-Plus，Controller 只做参数接收和响应封装，业务规则放到 Service。
- MySQL 是用户、商品、订单、消息、公告、审计等关键数据的事实源。
- Redis 只作为验证码、缓存、计数、限流、登录状态辅助，不以 Redis 单独决定订单或商品最终状态。
- 订单、商品审核、商品上下架等状态变更必须通过明确状态机控制。
- 所有接口统一响应结构、统一异常处理、统一参数校验，避免前端自行猜错误原因。
- 后续后端实现必须参考 `docs/后端工业级业务事务与高并发全景设计.md` 中的事务、缓存、幂等、审计和降级原则。

## 3. 系统角色

- 游客：浏览首页、商品列表、商品详情，可搜索筛选商品。
- 普通用户：注册登录、维护个人资料和地址、收藏商品、购买商品、发送留言、查看通知。
- 卖家：发布商品、管理自己的商品、处理订单发货、回复买家留言。
- 管理员：用户启用/禁用、商品审核、违规下架、订单异常处理、公告发布、统计查看。

## 4. 后端模块设计

### 4.1 `auth` 认证模块

能力范围：

- 注册：支持手机号或邮箱形式的账号注册，验证码可使用 Redis 模拟。
- 登录：用户名/手机号/邮箱 + 密码登录，成功后签发 JWT。
- 找回密码：验证码校验后重置密码。
- 退出登录：把 token 标识写入 Redis 黑名单或维护短期失效记录。

关键设计：

- 密码使用 BCrypt 存储，不保存明文。
- 登录失败按 IP 或账号维度做 Redis 限流。
- JWT 只保存用户 id、角色、过期时间，不保存敏感信息。

### 4.2 `user` 用户模块

能力范围：

- 个人中心：昵称、手机号、邮箱、头像、实名状态、信用分。
- 头像上传：先支持本地上传目录，数据库保存访问路径。
- 地址管理：新增、编辑、删除、设置默认地址。
- 用户信用和实名：答辩阶段可做可视化状态和后台审核入口。

关键设计：

- 用户资料和地址都落 MySQL。
- 普通用户只能修改自己的资料和地址。
- 管理员可查看用户列表、启用/禁用账号、调整角色。

### 4.3 `product` 商品模块

能力范围：

- 商品发布：标题、分类、价格、成色、富文本描述、多图上传。
- 商品浏览：首页推荐、分类筛选、关键词搜索、价格排序、时间排序、热度排序。
- 商品详情：图片轮播、卖家信息、收藏、浏览量、商品状态。
- 商品状态：待审核、在售、已驳回、已下架、已售出、已删除。
- 卖家管理：上架、下架、删除、编辑商品。

关键设计：

- 商品主信息、图片、收藏关系落 MySQL。
- 商品详情可使用 Redis 缓存，写操作提交后删除相关缓存。
- 浏览量可先写 Redis 计数，再在查询或后台任务中回写 MySQL。
- 商品审核通过后才能在前台列表展示。

### 4.4 `order` 订单模块

能力范围：

- 下单：支持直接购买，购物车作为可选增强。
- 模拟支付：订单从待付款变为待发货。
- 发货：卖家录入校内自提点、快递单号或备注。
- 收货：买家确认收货后订单完成。
- 取消：待付款或待发货阶段可取消，按规则恢复商品状态。
- 物流：查询模拟物流信息。

状态机：

```text
PENDING_PAYMENT -> PAID -> SHIPPED -> COMPLETED
PENDING_PAYMENT -> CANCELED
PAID -> CANCELED
```

关键设计：

- 下单、支付、发货、收货、取消必须在 Service 中校验当前状态。
- 订单金额以商品当前价格和后端计算为准，不能信任前端提交金额。
- 订单状态变化写入订单日志，便于后台追踪。
- 商品售出后不能被重复下单。

### 4.5 `message` 消息模块

能力范围：

- 买卖双方站内留言。
- 商品审核结果通知。
- 订单状态变更通知。
- 系统公告通知。
- 已读/未读标记。

关键设计：

- 普通留言落 `messages`。
- 系统通知落 `notifications` 或与 messages 通过类型区分。
- 未读数可用 Redis 缓存，最终以 MySQL 为准。

### 4.6 `admin` 后台模块

能力范围：

- 用户管理：列表、搜索、禁用、启用、角色分配。
- 商品审核：待审核列表、通过、驳回、违规下架。
- 订单管理：订单查询、异常处理、状态查看。
- 统计面板：注册用户量、商品发布量、成交量、成交金额、分类占比。
- 公告管理：发布公告、下线公告、前台展示。

关键设计：

- 后台操作必须校验管理员角色。
- 审核、禁用、违规下架、订单异常处理写审计日志。
- 统计优先使用 MySQL 聚合，后续可加入 Redis 或定时任务优化。

### 4.7 扩展模块

答辩阶段做轻量可演示能力：

- 推荐：按用户最近浏览分类、收藏分类、商品热度推荐。
- 敏感词：商品发布和留言提交时做本地词表过滤。
- 举报：用户举报商品或留言，管理员后台处理。

这些能力不作为第一轮核心事务链路，但需要在页面和接口中有可讲的入口。

## 5. 数据库设计方向

在现有 `sql/schema.sql` 基础上补齐或调整以下表：

- `users`
- `user_addresses`
- `categories`
- `products`
- `product_images`
- `favorites`
- `orders`
- `order_logs`
- `messages`
- `notifications`
- `announcements`
- `audit_logs`
- `reports`
- `sensitive_words`

关键字段约定：

- 业务表保留 `created_at`、`updated_at`。
- 关键状态表保留 `status` 和必要状态日志。
- 商品、订单等高并发修改对象保留 `version` 字段，为后续乐观锁留边界。
- 删除类操作优先软删除或状态变更，避免答辩演示时数据突然消失导致链路断开。

## 6. Redis 设计方向

第一轮需要实际落地的 Redis 场景：

- `captcha:register:{account}`：注册验证码。
- `captcha:reset:{account}`：找回密码验证码。
- `product:detail:{productId}`：商品详情缓存。
- `product:view:{productId}`：商品浏览量计数。
- `product:hot`：热门商品排行。
- `rate:login:{ip}`：登录限流。
- `jwt:blacklist:{tokenId}`：退出登录 token 黑名单。

缓存一致性原则：

- 商品、订单、用户资料写操作以 MySQL 提交为准。
- 写操作成功后删除相关 Redis 缓存。
- Redis 不可用时，核心读写可降级到 MySQL；验证码和限流能力可返回明确错误或走开发模式兜底。

### 7.2 页面范围

- 登录页：账号密码登录、注册入口、找回密码入口。
- 注册页：账号、手机号/邮箱、验证码、密码。
- 首页：推荐商品、热门商品、公告、分类入口。
- 商品列表：搜索、分类筛选、排序。
- 商品详情：图片轮播、卖家信息、收藏、立即购买、留言。
- 发布商品：多图上传、分类、价格、富文本描述。
- 个人中心：资料、头像、地址、我的商品、我的收藏。
- 订单中心：买家订单和卖家订单，支持推进状态。
- 消息中心：留言、系统通知、已读未读。
- 后台管理：用户、商品审核、订单、公告、统计。

### 7.3 UI 原则

- 首屏直接进入应用，不做营销页。
- 管理后台采用操作台风格，信息密度适中，便于答辩演示。
- 商品页面突出图片、价格、分类、状态和操作按钮。
- 所有状态使用统一标签组件，避免页面到处手写状态样式。

## 8. 接口设计方向

接口前缀统一使用 `/api`。

核心接口示例：

```text
POST   /api/auth/register
POST   /api/auth/login
POST   /api/auth/reset-password
GET    /api/users/me
PUT    /api/users/me
GET    /api/users/me/addresses
POST   /api/products
GET    /api/products
GET    /api/products/{id}
POST   /api/products/{id}/favorite
POST   /api/orders
POST   /api/orders/{id}/pay
POST   /api/orders/{id}/ship
POST   /api/orders/{id}/confirm
POST   /api/orders/{id}/cancel
GET    /api/messages
POST   /api/messages
POST   /api/messages/{id}/read
GET    /api/admin/dashboard
GET    /api/admin/products/pending
POST   /api/admin/products/{id}/audit
POST   /api/admin/announcements
```

统一响应沿用当前 `ApiResponse`，后续补错误码和分页结构。

## 9. 错误处理与安全

- 使用统一异常处理，将业务异常转换为明确错误码和错误消息。
- 请求参数使用 Bean Validation 校验。
- 登录态通过 JWT 解析当前用户，不信任前端传入的 `userId`、`sellerId`、`buyerId`。
- 管理员接口通过角色校验。
- 图片上传限制文件大小、文件类型和保存目录。
- 敏感词过滤用于商品标题、描述、留言内容。
- 后台关键操作写审计日志。

## 10. 测试与验收

后端验收：

- `mvn -f backend/pom.xml test` 通过。
- 覆盖认证、商品、订单、消息、后台核心接口的 MockMvc 测试。
- 至少验证订单非法状态流转会失败。
- 至少验证普通用户不能访问后台接口。

前端验收：

- `pnpm test` 通过。
- `pnpm build` 通过。
- 登录、商品列表、商品详情、下单、订单状态推进、后台审核都有基础组件测试或页面渲染测试。

人工验收主线：

1. 注册/登录演示用户。
2. 发布一个商品。
3. 管理员审核通过。
4. 首页或列表搜索到商品。
5. 进入详情收藏并下单。
6. 模拟支付。
7. 卖家发货。
8. 买家确认收货。
9. 查看消息通知和订单状态。
10. 后台查看统计和公告。

## 11. 实施阶段

### 阶段一：结构升级

- 前端拆路由、API、组件、视图、store。
- 后端建立 auth、user、product、order、message、admin 包。
- 保留 `/api/demo/**` 作为兜底演示接口，避免中途项目不可运行。

### 阶段二：用户与认证

- 实现注册、登录、JWT、找回密码模拟、个人资料、地址。
- Redis 落地验证码和登录限流。

### 阶段三：商品模块

- 商品发布、多图、分类、搜索筛选排序、详情、收藏、审核、上下架。
- Redis 落地商品详情缓存、浏览量、热门榜。

### 阶段四：订单交易

- 直接购买、订单状态机、模拟支付、发货、确认收货、取消、物流信息。
- 写订单日志和必要审计。

### 阶段五：消息与后台

- 站内留言、系统通知、已读未读。
- 后台用户、商品审核、订单异常、公告、统计。

### 阶段六：扩展与文档

- 推荐、敏感词、举报轻量落地。
- 补需求分析、系统设计、数据库设计、接口文档、测试报告、用户说明和答辩路线。

## 12. 非目标

- 不接入真实支付平台，只做沙箱式模拟支付。
- 不接入真实短信、邮件或 OSS，验证码和上传先使用本地模拟。
- 不强制引入 MQ，若时间不足，以本地事件和审计日志替代；MQ/outbox 作为后续增强。
- 不在第一轮追求完整压测平台，只保留接口测试和基础性能说明。

## 13. 成功标准

- 项目可以在本机稳定启动前后端。
- MySQL 中能看到用户、商品、订单、消息、公告等核心业务数据。
- Redis 至少被验证码、商品缓存或浏览量计数实际使用。
- 答辩主线可以完整演示：注册登录、发商品、审核、搜索、详情、下单、支付、发货、收货、通知、后台统计。
- 代码结构比当前单页/单服务演示版明显清晰，可解释每个模块职责。
- README 和文档能告诉组员如何启动、如何测试、如何演示。
