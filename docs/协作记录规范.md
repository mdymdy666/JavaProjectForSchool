# 协作记录规范

## 背景

本项目可能由多名成员和多个 Agent 协作推进。为避免修改不可追踪、上下文丢失或重复工作，需要维护统一的协作记录。

## 目标

通过根目录 `codex.md` 记录每次重要代码、文档或环境配置变动，保证后续成员能快速理解项目状态。

## 记录文件

Agent 开始工作前应检查项目中是否存在已有记录文件或记录规范，常见位置包括：

```text
codex.md
.agent/doc/codex.md
.agents/doc/codex.md
gpt.md
claude.md
trae.md
deepseek.md
```

如果项目已有明确记录规范，应优先遵循已有规范，避免重复创建冲突记录。

当前项目采用根目录：

```text
codex.md
```

## 提示词生成类任务规则

当用户要求 Agent 生成、优化或改写 AI 提示词时，Agent 的主要职责是把用户原始想法整理成更清晰、更完整、更适合 AI / Agent 执行的提示词，而不是默认直接替用户完成提示词中的任务。

提示词生成类任务应遵循：

1. 保留用户原始意图、核心任务、限制条件和验收目标。
2. 将模糊需求拆成可执行步骤，补充必要的背景、目标、约束、输出格式、验收标准和自检要求。
3. 内容必须严格基于当前项目已有信息，不捏造项目已经完成的功能或不存在的文件。
4. 第一次处理项目相关提示词时，应浏览项目结构、关键文档和当前 Git 状态；后续处理时，应关注项目变化。
5. 如果提示词要求其他 Agent 输出文件，应通过提示词约束输出文件数量、位置、命名和内容范围，避免生成过多、重复或偏离项目的文档。
6. 如果关键信息无法合理推断，应在生成的提示词中要求执行 Agent 用最少问题向用户确认。
7. 生成的提示词应可直接复制使用，不只给宽泛建议。

提示词中可按任务需要包含简洁的协作记录规则，但协作记录规则不应盖过用户任务本身。

## 记录时机

以下变动后应记录：

1. 代码变动。
2. 文档变动。
3. 环境配置变动。
4. 依赖新增、升级或移除。
5. Git 仓库初始化、目录结构调整、脚本新增。
6. 用户明确要求记录的其他事项。

如果用户明确说“不需要记录”，才可以跳过记录。

代码修改、配置修改、脚本修改等会影响项目行为的变动，默认必须记录到 `codex.md`；除非用户明确说明“不需要记录改动记录”。

## 记录单位

记录应以 Git 变动为单位。一次相关变动作为一条记录，不把无关事项混在同一条中。

## 记录字段

每条记录应包含：

| 字段 | 说明 |
|---|---|
| 时间 | 建议使用 `YYYY-MM-DD HH:mm` |
| 执行 Agent | 例如 Codex |
| 变动类型 | 初始化、文档、代码、环境、测试、修复等 |
| 涉及文件 | 列出主要文件 |
| 完成内容 | 简明说明做了什么 |
| 验证方式 | 命令、人工检查或未验证原因 |
| 后续状态 | 已完成、待确认、待实现、阻塞等 |

## 记录模板

```markdown
## YYYY-MM-DD HH:mm - 标题

- 执行 Agent：
- 变动类型：
- 涉及文件：
- 完成内容：
- 验证方式：
- 后续状态：
```

## 验收标准

- 重要变动在 `codex.md` 中可追踪。
- 记录与 Git 变动范围一致。
- 记录不包含敏感信息。
- 提示词生成类任务应输出可直接执行的提示词，并覆盖任务背景、目标、步骤、约束、输出格式、验收标准和自检要求。
- 项目相关提示词应基于项目真实状态，不把计划内容写成已完成内容。
