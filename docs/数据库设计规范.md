# 数据库设计规范

## 背景

指导书要求项目使用 MySQL，并设计用户、商品、订单、消息、分类等核心数据表。数据库设计质量会影响交易流程、查询性能、数据一致性和后续扩展。

## 目标

统一数据库设计文档格式，保证表结构、字段、索引、约束和表关系可追踪、可评审。

## 范围

适用于后续 MySQL 数据库设计、初始化脚本、变更脚本和 ER 图说明。

## 表设计文档要求

每张表应至少包含：

| 内容 | 要求 |
|---|---|
| 表名 | 使用清晰稳定的英文小写命名 |
| 字段名 | 使用英文小写和下划线 |
| 类型 | 明确 MySQL 字段类型和长度 |
| 是否为空 | 标明 `NOT NULL` 或可为空 |
| 默认值 | 标明默认值或无默认值 |
| 索引 | 主键、唯一索引、普通索引、联合索引 |
| 约束 | 外键、唯一约束、业务约束 |
| 字段说明 | 说明字段含义和取值范围 |
| 表关系 | 说明与其他表的一对一、一对多、多对多关系 |

## 推荐模板

```markdown
## 表：表名

### 用途

说明该表存储的业务数据。

### 字段

| 字段名 | 类型 | 是否为空 | 默认值 | 索引 | 约束 | 说明 |
|---|---|---|---|---|---|---|

### 表关系

| 关联表 | 关系 | 关联字段 | 说明 |
|---|---|---|---|

### 设计说明

- 
```

## 设计约束

1. 用户密码后续必须加密存储，计划使用 BCrypt 等安全散列方案。
2. 订单状态、商品状态、审核状态等枚举值应在文档中统一定义。
3. 涉及订单创建、状态变更、库存或商品状态更新时，应考虑事务一致性。
4. 高频查询字段应评估索引，例如商品分类、状态、发布时间、卖家 ID、订单用户 ID。
5. 不在仓库中提交真实生产数据或本机数据库数据目录。
6. SQL 脚本后续建议放入 `sql/` 目录。

## 验收标准

- 每张表的字段、索引、约束和关系完整清晰。
- 表设计能支撑指导书要求的用户、商品、订单、消息、后台管理等模块。
- 数据库文档与实际 SQL 脚本保持一致。
