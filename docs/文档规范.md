# 文档规范

## 背景

本项目来自《徐士伟-软件与理论综合设计与实践项目指导书【徐士伟-题目2-2026】》，要求完成基于 Vue3 + Spring Boot + MySQL + Redis 的校园二手物品交易平台。项目文档是需求分析、系统设计、接口约定、数据库设计、测试和答辩准备的重要依据。

## 目标

建立统一、清晰、可长期维护的文档规范，保证后续成员和 Agent 在同一口径下记录需求、设计、实现和测试信息。

## 范围

本规范适用于项目中的需求文档、设计文档、接口文档、数据库文档、测试文档、环境文档、协作记录和后续开发说明。

## 存放位置

除根目录必需文件外，所有项目文档统一放在 `docs/` 目录。

根目录允许保留：

- `README.md`
- `codex.md`
- `LICENSE`
- 项目构建文件，如后续创建的 `pom.xml`、`package.json`

## 命名规范

当前 `docs/` 目录采用中文名称，名称应稳定、清晰、可直接表达主题，例如：

- `项目说明.md`
- `开发环境.md`
- `接口文档规范.md`
- `数据库设计规范.md`

如后续新增英文文档，应使用 kebab-case，例如 `api-design.md`。同一目录内应避免同一主题同时存在中文名和英文名的重复文档。

## 文档结构要求

项目文档建议至少包含以下部分：

| 部分 | 要求 |
|---|---|
| 背景 | 说明文档产生原因和项目上下文 |
| 目标 | 说明文档要解决的问题 |
| 范围 | 说明适用边界和不包含内容 |
| 具体内容 | 写明需求、设计、规则、流程或数据 |
| 约束 | 写明技术、环境、安全、协作限制 |
| 验收标准 | 写明如何判断文档内容可用 |
| 更新记录 | 记录重要修改时间、修改人和变更内容 |

## 写作规则

1. 严格区分“项目要求”“计划实现”“已实现”。
2. 当前项目尚未创建前后端业务代码，不得在文档中描述为“功能已完成”。
3. 依赖版本必须来自命令检测、配置文件或用户确认，不得凭空填写。
4. 涉及账号、密码、密钥、数据库连接串时，只写配置项名称和示例占位符，不写真实值。
5. 接口、数据库、测试等文档应随着实现同步更新。

## 后续文档更新规则

1. 需求变更后，同步更新项目说明、接口文档和数据库设计文档。
2. 新增接口后，同步更新接口文档和测试用例。
3. 新增表或字段后，同步更新数据库设计文档和初始化 SQL。
4. 新增依赖后，同步更新 `开发环境.md` 或 `依赖版本清单.md`。
5. 每次重要文档变动后，在根目录 `codex.md` 记录本次 Git 变动单元。

## 验收标准

- 文档可在 `docs/` 目录中按主题找到。
- 文档命名清晰，没有同主题冲突文件。
- 文档内容不夸大项目状态，不把计划功能写成已实现功能。
- 文档包含必要的背景、目标、范围、约束、验收标准和更新记录。
