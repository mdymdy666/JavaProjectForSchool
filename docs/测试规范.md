# 测试规范

## 背景

指导书要求项目进行前后端接口联调、单元测试、集成测试、接口测试、压力测试与性能优化。测试文档应覆盖核心业务流程与关键非功能需求。

## 目标

建立统一测试记录格式，保证缺陷可复现、测试结果可追踪、验收标准可判断。

## 范围

适用于后续后端单元测试、前端组件测试、接口测试、集成测试、性能测试和验收测试。

## 测试文档要求

测试文档应包含：

| 内容 | 要求 |
|---|---|
| 测试范围 | 明确测试模块、功能点和边界 |
| 测试环境 | 记录操作系统、Java、Node、MySQL、Redis、浏览器等版本 |
| 测试数据 | 说明测试账号、商品、订单、消息等数据准备方式，不写真实敏感信息 |
| 测试用例 | 包含前置条件、操作步骤、输入数据 |
| 预期结果 | 明确应返回或展示的结果 |
| 实际结果 | 记录真实执行结果 |
| 缺陷记录 | 记录缺陷编号、现象、复现步骤、严重程度、修复状态 |

## 推荐模板

```markdown
## 测试项

- 模块：
- 测试类型：
- 测试环境：
- 前置条件：

| 用例编号 | 用例名称 | 操作步骤 | 测试数据 | 预期结果 | 实际结果 | 状态 | 备注 |
|---|---|---|---|---|---|---|---|
```

## 后续测试重点

1. 用户注册、登录、JWT 认证与权限控制。
2. 商品发布、多图上传、富文本内容、商品搜索与筛选。
3. 订单创建、状态流转、取消、完成、异常处理。
4. 买卖双方消息沟通和系统通知。
5. 后台用户管理、商品审核、订单管理、公告管理。
6. Redis 缓存策略、验证码、热点商品等场景。
7. XSS、CSRF、SQL 注入、防刷等安全场景。
8. 响应式界面和主流 PC 浏览器兼容性。

## Smoke Test 规范

本项目允许在 `test/` 目录创建独立 smoke test，用于验证环境和配置型依赖是否可解析、编译、测试、构建或短启动。

约束：

1. smoke test 必须放在 `test/` 目录下。
2. 不得把 smoke test 写成正式前端或正式后端业务工程。
3. 不得在 smoke test 中实现校园二手交易业务逻辑。
4. 不得提交真实密码、密钥、`.env` 文件、数据库数据、Redis 数据。
5. 如需保留 smoke test，应只保留源文件、配置文件、测试脚本、锁文件和测试报告。
6. 不提交 `node_modules/`、`target/`、`dist/`、Maven 缓存、pnpm store 或日志文件。

当前仓库已按要求删除 `test/` 目录；如后续需要重新验证依赖，可按本节规范重新创建独立 smoke test。

## 环境依赖检测脚本

当前仓库提供 Windows 环境依赖检测脚本：

```text
scripts/check-dependencies.bat
```

用途：

1. 检查 Git、Java、Javac、Maven、Node.js、npm、pnpm、MySQL、Redis、VS Code、IntelliJ IDEA、Apifox 等本机工具。
2. 兼容不同安装路径，不因目录不同直接判定失败。
3. 识别版本差异，并输出“已对齐 / 可接受待确认 / 需调整 / 未检测到 / 项目未配置 / 检测失败”。
4. 后续如果项目创建正式 `pom.xml` 或 `package.json`，脚本会检测 Spring Boot、Vue3、Vite 等配置型依赖。
5. 生成检测报告：`reports/dependency-check-report.md`。

约束：

1. 脚本只做检测，不安装、卸载或重装任何软件。
2. 脚本不修改系统 PATH、Java、MySQL 或 Redis 配置。
3. 脚本不猜测 MySQL/Redis 密码，不重置账号，不删除服务或数据。
4. `reports/dependency-check-report.md` 可作为最近一次检测报告保留；`reports/tmp/` 不提交。

## 验收标准

- 核心业务流程有测试用例。
- 测试环境和依赖版本可追踪。
- 缺陷记录能支持复现和回归。
- 测试结果与接口文档、数据库设计保持一致。
