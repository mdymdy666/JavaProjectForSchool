# 环境安装与版本对齐

## 适用对象

本文档给参与“校园二手物品交易平台的设计与实现”的组员使用，用于从零准备开发环境或检查已有环境是否符合项目要求。

本文档是组员环境准备的唯一对外说明。不要照抄某台电脑的历史安装路径、旧服务名或本机迁移记录。

## 已安装其他版本或其他目录时如何处理

组员电脑上已经装了必须软件时，优先检查可用性和版本兼容性，不要一上来卸载、重装或搬目录。处理顺序如下：

1. 先记录现状：版本、安装路径、PATH 命令来源、服务名、服务状态。
2. 再判断是否可接受：能在新 PowerShell 中执行验证命令，且版本与项目推荐版本一致或兼容。
3. 如果只是安装目录不同：不需要移动软件，只需要把实际路径写入检查表；PATH 能正常解析即可。
4. 如果版本不同但兼容：可临时保留，并在检查表中标为“版本不同，待组内确认”。
5. 如果版本不同且风险较高：不要直接卸载旧版本，建议并行安装推荐版本，并通过环境变量或 PATH 让本项目优先使用推荐版本。
6. 如果 PATH 指向了错误版本：优先调整用户 PATH 的顺序，或在本项目启动脚本中指定完整路径；不要改系统级 PATH，除非组员明确知道影响范围。
7. 如果 MySQL 或 Redis 已有旧服务：不要删除、重建或改密码；先记录服务名、端口、版本、运行状态，再决定本项目使用哪个服务。

### 版本差异处理建议

| 软件 | 可接受情况 | 建议处理 |
|---|---|---|
| Git | 2.4x 或更高，命令可用 | 通常可保留；记录实际版本 |
| JDK | Java 21 LTS 或 Java 24，`java` 与 `javac` 版本一致 | 推荐全组最终统一；短期可记录差异后组内确认 |
| Maven | 3.9.x | 通常可保留；低于 3.8 建议升级 |
| Node.js | 22.x，npm 可用 | 通常可保留；低于 20 建议升级 |
| npm | 随 Node.js 安装，命令可用 | 通常不单独强制对齐 |
| pnpm | 11.7.0 优先 | 不一致时建议用 Corepack 或 npm 固定到 11.7.0 |
| MySQL | 8.0.x 优先 | 8.0.x 通常可接受；9.x 或 5.x 需组内确认，避免 SQL/驱动/字符集差异 |
| Redis | 8.0.3 优先 | 7.x/8.x 可先记录；正式联调前统一确认 |
| Apifox | 2.7.44 或相近版本 | 可保留；确保能打开项目接口集合 |
| IDEA / VS Code | 能正常启动 | 不强制目录和小版本 |

### 目录差异处理建议

安装目录不是项目标准的一部分。团队只要求：

1. 每台电脑能清楚说出实际安装路径。
2. 新 PowerShell 中验证命令可执行。
3. `JAVA_HOME`、`MAVEN_HOME` 等变量指向真实存在的目录。
4. PATH 中优先命中本项目要使用的版本。
5. 不把本机安装目录、运行时目录、数据库数据目录放进项目仓库。

例如，下面两种都可以接受：

```text
D:\dev-tools\java\jdk-24.0.2
C:\Program Files\Java\jdk-24
```

只要 `java -version` 和 `javac -version` 输出符合要求，并且检查表写清楚来源即可。

## 推荐安装目录

优先使用软件官方默认安装目录。

如果希望统一自定义目录，推荐使用：

```text
D:\dev-tools\软件名称\版本号
```

示例：

```text
D:\dev-tools\java\jdk-24.0.2
D:\dev-tools\maven\apache-maven-3.9.11
D:\dev-tools\nodejs\22.16.0
D:\dev-tools\mysql\mysql-8.0.20-winx64
D:\dev-tools\redis\8.0.3
D:\dev-tools\apifox\2.7.44
```

如果没有 `D:` 盘，可以使用 `C:\dev-tools` 或 `E:\dev-tools`。重点不是盘符一致，而是版本、环境变量、PATH 和验证命令一致。

## 环境变量建议

| 变量名 | 示例值 | 说明 |
|---|---|---|
| `JAVA_HOME` | `D:\dev-tools\java\jdk-24.0.2` | 指向 JDK 根目录 |
| `MAVEN_HOME` | `D:\dev-tools\maven\apache-maven-3.9.11` | 指向 Maven 根目录 |
| `MYSQL_HOME` | `D:\dev-tools\mysql\mysql-8.0.20-winx64` | 可选，指向 MySQL 根目录 |
| `REDIS_HOME` | `D:\dev-tools\redis\8.0.3` | 可选，指向 Redis 根目录 |
| `NODE_HOME` | `D:\dev-tools\nodejs\22.16.0` | 可选，指向 Node.js 根目录 |

## PATH 建议

如果使用 `D:\dev-tools` 目录，可将以下路径加入用户 PATH。若使用官方默认目录，请按实际安装路径调整。

```text
D:\dev-tools\git\2.49.0\cmd
D:\dev-tools\java\jdk-24.0.2\bin
D:\dev-tools\maven\apache-maven-3.9.11\bin
D:\dev-tools\nodejs\22.16.0
D:\dev-tools\mysql\mysql-8.0.20-winx64\bin
D:\dev-tools\redis\8.0.3
```

Apifox、IntelliJ IDEA 通常不要求加入 PATH，可以通过快捷方式启动。VS Code 建议安装时勾选命令行启动支持，便于使用：

```powershell
code .
```

## 推荐安装顺序

1. 安装 Git。
2. 安装 JDK，配置 `JAVA_HOME` 和 PATH。
3. 安装 Maven，配置 `MAVEN_HOME` 和 PATH。
4. 安装 Node.js，确认 npm 可用。
5. 安装或启用 pnpm，并固定到 `11.7.0`。
6. 安装 MySQL 8.0.20，确认 MySQL 服务可启动，客户端命令可用。
7. 安装 Redis 8.0.3，确认服务端和客户端命令可用。
8. 安装 IntelliJ IDEA、VS Code。
9. 安装 Apifox。
10. 打开新终端执行验证命令。

## pnpm 安装方式

以下方式二选一即可。

方式一：Corepack。

```powershell
corepack enable
corepack prepare pnpm@11.7.0 --activate
pnpm --version
```

方式二：npm 全局安装。

```powershell
npm install -g pnpm@11.7.0
pnpm --version
```

不要同时维护多套 pnpm 来源，避免不同终端版本不一致。

## 自动检测脚本

仓库提供 Windows 自动检测脚本：

```powershell
scripts\check-dependencies.bat
```

脚本用于处理“不同组员安装路径不同、版本不同”的情况。它会按以下顺序尝试发现依赖：

1. PATH 中的命令。
2. `JAVA_HOME`、`MAVEN_HOME`、`MYSQL_HOME`、`REDIS_HOME`、`NODE_HOME` 等环境变量。
3. `D:\dev-tools`、`C:\dev-tools`、`D:\soft_wares`、`D:\soft_wares2`、`D:\soft_wares5`、`C:\Program Files` 等常见目录。
4. MySQL / Redis 的 Windows 服务或进程。
5. 如果后续创建了正式 `pom.xml` 或 `package.json`，会检查其中的 Spring Boot、Spring Security、MySQL JDBC Driver、Spring Data Redis、Vue3、Vite、Axios、Vitest 等配置型依赖。

运行后生成报告：

```text
reports/dependency-check-report.md
```

状态含义：

| 状态 | 含义 |
|---|---|
| 已对齐 | 检测版本与团队推荐版本一致 |
| 可接受待确认 | 版本不同或未通过 PATH 命中，但大概率可用，需要组内确认 |
| 需调整 | 版本过旧、主版本差异较大，或 PATH 指向明显不适合的版本 |
| 未检测到 | PATH、环境变量、常见目录、服务均未发现 |
| 项目未配置 | 当前尚未创建正式配置文件，因此跳过配置型依赖检测 |
| 检测失败 | 找到候选项，但命令执行或依赖解析失败 |

脚本不会：

1. 修改系统 PATH。
2. 安装、卸载或重装任何软件。
3. 修改 Java、MySQL、Redis 配置。
4. 重置 MySQL 密码或创建/删除数据库。
5. 修改 Redis 密码或删除 Redis 数据。
6. 写入真实账号、密码、密钥或 `.env`。

## 常见问题

### 命令安装了但终端识别不到

先关闭当前终端，重新打开 PowerShell。环境变量和 PATH 修改后，旧终端不会自动刷新。

### `mysql --version` 不可用

检查 MySQL 的 `bin` 目录是否加入 PATH，例如：

```text
D:\dev-tools\mysql\mysql-8.0.20-winx64\bin
```

### `pnpm --version` 不对

检查是否存在多套 pnpm。建议保留一种方式，然后重新安装或激活：

```powershell
npm install -g pnpm@11.7.0
```

或：

```powershell
corepack prepare pnpm@11.7.0 --activate
```

### Java 版本是否必须是 24.0.2

当前团队对齐版本是 Java 24.0.2。如果后续课程、部署或 Spring Boot 版本要求使用 Java 21 LTS，需要全组统一切换，并同步修改文档。

## 验收标准

1. 必须软件均已安装。
2. 验证命令能在新 PowerShell 中执行。
3. 版本与本文档一致，或经组内确认可接受。
4. MySQL 服务可启动，客户端可连接。
